# Typst as a rendering target

This document explores adding Typst output to packages generated by
`create_indiedown_package()`. **Nothing here is implemented yet** — it is
a design note. The deeper analysis of the LaTeX→Typst translation lives
in [`cynkradown/TYPST.md`](https://github.com/cynkra/cynkradown/blob/main/TYPST.md)
since that is the reference implementation; this file focuses on the
scaffolder.

## What indiedown would need to change

The current skeleton at `inst/mypackage/` ships two rendering surfaces:

1. **R Markdown via rmarkdown** — `inst/indiedown/{preamble.tex,
   default.yaml, pre_processor.R}` + `inst/rmarkdown/templates/report/`.
2. **Quarto via the LaTeX path** —
   `inst/quarto/_extensions/mypackage/mypackage/` + `R/use_quarto.R`
   (added in the previous round).

A typst surface would be a third: `inst/quarto/_extensions/mypackage/
mypackage-typst/` with `typst-template.typ` and `typst-show.typ`. The
scaffolder needs the same treatment we just added for the LaTeX Quarto
extension — string substitution on the literal `mypackage`, directory
rename to the user's package name.

## Concrete changes to `create_indiedown_package()`

1. Add files to `inst/mypackage/inst/quarto/_extensions/mypackage/mypackage-typst/`:
   - `_extension.yml` — `contributes.formats.typst` with
     `template-partials: [typst-template.typ, typst-show.typ]` and
     `format-resources: [fonts, res]`.
   - `typst-template.typ` — defines `#let mypackage-article(...) = { ... }`
     with the same parameters the LaTeX preamble currently exposes (font,
     fontsize, lang, sectionnumbering, twocolumn, wide). Default body
     can be very plain so users have a clear starting point — the heavy
     styling lives in downstream packages like cynkradown.
   - `typst-show.typ` — pandoc partial that maps metadata to a
     `#show: doc => mypackage-article(...)` call.
2. Extend the substitution list in `create_indiedown_package()`:
   ```r
   files <- c(
     # ... existing ...
     "inst/quarto/_extensions/mypackage/mypackage-typst/_extension.yml",
     "inst/quarto/_extensions/mypackage/mypackage-typst/typst-template.typ",
     "inst/quarto/_extensions/mypackage/mypackage-typst/typst-show.typ"
   )
   ```
   The directory rename block already in place needs an extra pair of
   `fs::file_move()` calls for the `-typst` variant.
3. Extend `R/use_quarto.R` (`use_mypackage_quarto()`) with a `format`
   argument:
   ```r
   use_mypackage_quarto(format = c("pdf", "typst"))
   ```
   so users can opt into either surface or both.
4. Add `inst/mypackage/inst/quarto/examples/example-typst.qmd`
   alongside the existing `example.qmd`, with `format:
   mypackage-typst` in its YAML header.
5. Bump the test snapshot at
   `tests/testthat/_snaps/create_indiedown_package.md` to include the
   new files.

## Things that stay the same

- The rmarkdown surface (`inst/indiedown/` + the R Markdown template) is
  not touched. Both new and existing downstream packages keep that
  working.
- The R helper functions in the skeleton (`cd_page_title`,
  `cd_format_date`, `cd_knit_chunk_opts`, `default`, `indiedown_glue`)
  are output-format-agnostic — they don't emit LaTeX directly, so they
  do not need a typst variant.
- `inst/indiedown/preamble.tex` and the indiedown documentation about
  customization points stay as-is. The vignettes pick up an extra
  section about the typst surface (parallel to what we just added
  about the LaTeX Quarto extension).

## What this enables for downstream packages

cynkradown is the motivating example. Once the indiedown skeleton ships
both surfaces, packages built on top inherit a working typst scaffold
they can customize, just as they currently inherit the LaTeX one. The
heavy lifting — porting `\definecolor`, `\titleformat`, `\fancyhdr`,
TikZ cover pages, etc. — lives in those downstream packages, not in
indiedown itself.

## Suggested ordering

Do cynkradown's typst port first (see `cynkradown/TYPST.md`). Once that
is working and we understand the right shape of `typst-template.typ` /
`typst-show.typ` for our style of documents, generalize the **plain**
version of those files into the indiedown skeleton. Doing it in the
other order risks designing the skeleton against assumptions that don't
survive contact with the real cynkra preamble.
